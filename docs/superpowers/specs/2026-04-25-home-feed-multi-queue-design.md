# Home Feed Multi-Queue Design

## 1. Summary

Artech 홈 피드를 `긴 최신 기사 리스트 + 상단 랜덤 추천` 구조에서 `빠르게 고를 수 있는 멀티 큐 홈`으로 재설계한다.

이 설계의 핵심은 다음과 같다.

- 홈의 주 역할을 `많이 보여주기`가 아니라 `빨리 고르게 하기`로 바꾼다.
- 추천을 하나의 캐러셀로 두지 않고, 목적이 다른 여러 큐로 분리한다.
- 추천은 로컬 신호(읽기이력 + 즐겨찾기) 기반 개인화로 바꾸되, 최신성과 다양성을 함께 반영한다.
- `HomeViewModel`에 추천 로직을 계속 얹지 않고, 관심사 계산과 홈 큐 조합 책임을 분리한다.

## 2. Current State

현재 홈 피드 구현은 다음 특성을 가진다.

- `HomeViewModel`이 기사 로드, 랜덤 추천, 카테고리 필터, 안 본 글 필터, 스크롤 위치 저장, 추천 새로고침 횟수 관리까지 함께 담당한다.
- 추천은 `최신 20개` 범위 안에서 `랜덤 5개`를 뽑는 구조다.
- 추천 새로고침은 더 적합한 대체 후보를 고르는 방식이 아니라 랜덤 재추첨에 가깝다.
- 홈의 메인 탐색 방식은 긴 리스트 스크롤이다.
- 문서에는 검색과 pull-to-refresh가 정의되어 있지만, 현재 홈 구현과는 차이가 있다.

이 구조는 기사 수가 많아질수록 다음 문제가 커진다.

- 사용자가 읽을 글을 고르기 전에 많이 내려야 한다.
- 추천의 의미가 약하고, 왜 이 글이 추천인지 이해하기 어렵다.
- 최신 리스트가 홈 대부분을 차지해서 개인화가 체감되기 어렵다.
- 추천과 홈 상태 관리가 `HomeViewModel`에 과도하게 모인다.

## 3. Problem Statement

사용자는 홈에서 `무엇을 읽을지 빠르게 결정`하고 싶지만, 현재 구조는 `계속 내려보는 것`을 전제로 한다.

따라서 이번 설계는 다음 질문에 답해야 한다.

- 긴 스크롤 없이도 홈에서 읽을 글을 바로 고를 수 있는가?
- 개인화 추천이 단순 랜덤보다 더 납득 가능한 방식으로 보이는가?
- 최신 글 탐색은 유지하되, 홈의 중심 역할에서 내려올 수 있는가?
- 현재 코드 구조 안에서 추천 실험과 UX 개선을 지속 가능한 형태로 분리할 수 있는가?

## 4. Goals

- 홈을 `멀티 큐 탐색 허브`로 전환한다.
- 홈 첫 화면에서 스크롤보다 선택이 먼저 일어나게 만든다.
- 읽기이력과 즐겨찾기를 함께 사용하는 관심사 기반 추천을 도입한다.
- 추천 큐별 역할을 분리해 사용자가 각 섹션의 의미를 직관적으로 이해하게 만든다.
- 추천 조합 로직을 UI 상태 관리에서 분리해 테스트 가능하게 만든다.

## 5. Non-Goals

- 서버 기반 계정 프로필 개인화 도입
- 협업 필터링, 임베딩 기반 추천, 외부 랭킹 시스템 도입
- 전체 앱 네비게이션 구조 변경
- 블로그 화면, 즐겨찾기 화면, 읽기이력 화면의 동작 개편
- 홈에서 모든 탐색 문제를 한 번에 해결하려는 대규모 IA 재편

## 6. Chosen Product Direction

검토한 방향은 다음 세 가지였다.

1. 멀티 큐 홈
2. 기존 최신 피드 유지 + 추천 강화
3. 검색/필터 우선 홈

이번 설계는 `1. 멀티 큐 홈`을 채택한다.

이유는 다음과 같다.

- 현재 문제는 추천 정확도 부족보다 `탐색 방식이 스크롤 중심`이라는 데 더 가깝다.
- 추천을 강화해도 긴 최신 리스트 안에 머무르면 체감 개선이 제한적이다.
- 사용자는 `빠르게 고르기`를 우선했고, 검색보다 추천 큐 중심 진입을 원했다.

## 7. Home Information Architecture

홈은 단일 기사 리스트가 아니라, 목적이 다른 짧은 섹션들의 수직 스택으로 구성한다.

### 7.1 오늘 읽을 글

홈의 첫 섹션이다.

- 개인화 추천의 메인 엔트리
- 노출 수는 3개 내외
- 가장 빠르게 하나를 고를 수 있어야 함
- 같은 카테고리/블로그로 과도하게 몰리지 않도록 제한

### 7.2 관심 주제에서 고르기

기사 카드 자체보다 `관심 카테고리 진입점`을 제공하는 섹션이다.

- 상위 관심 카테고리 2~4개 노출
- 각 카테고리별 `안 읽은 글 수`를 함께 보여줌
- 사용자가 추천 카드가 맞지 않아도 빠르게 관심 분야로 이동할 수 있게 함

### 7.3 놓치기 쉬운 글

최신순 상단에서는 밀렸지만 취향과는 맞는 글을 복구하는 섹션이다.

- 안 읽은 글만 포함
- 최신 최상단에서 밀린 글을 다시 제안
- 개인화는 강하지만 최신성은 `오늘 읽을 글`보다 한 단계 낮게 반영

### 7.4 최신 글

최신 글은 홈의 보조 섹션으로 축소한다.

- 일부만 미리보기로 노출
- 긴 최신 탐색은 `LatestFeedScreen`으로 이동
- 홈의 메인 역할은 더 이상 최신 전체 피드가 아님

### 7.5 검색

검색 진입점은 유지한다. 다만 홈의 주 사용 흐름은 검색보다 추천 큐 중심이 된다.

## 8. Recommendation Model

추천은 `하나의 추천 리스트`를 만드는 대신, `같은 후보 기사 풀에서 서로 다른 목적의 큐를 조합`하는 방식으로 설계한다.

### 8.1 Candidate Pool

홈 추천용 후보 기사 풀은 현재보다 넓게 잡는다.

- 최신 20개만 사용하지 않음
- 최근 기사 약 80~120개 범위를 기본 후보로 사용
- 각 큐는 이 후보 풀에서 별도 규칙으로 선정

이 범위는 구현 단계에서 성능과 실제 기사량에 맞춰 튜닝할 수 있지만, 설계 기준은 `최신 20개보다 확실히 넓은 풀`이다.

### 8.2 Interest Signals

관심사 계산은 로컬 데이터만 사용한다.

- 읽기이력: 약한 관심 신호
- 즐겨찾기: 강한 관심 신호
- 최근 행동일수록 높은 가중치

관심사는 최소 두 축으로 계산한다.

- 카테고리 선호
- 블로그 선호

### 8.3 Scoring Principles

각 후보 기사에는 다음 성격의 점수를 합산한다.

- 카테고리 관심도
- 블로그 관심도
- 읽지 않은 기사 보너스
- 너무 오래되지 않은 기사 보정
- 다양성 제약 보정

설계 기준상 중요 우선순위는 다음과 같다.

1. 읽지 않은 글이어야 한다.
2. 사용자 관심사와 맞아야 한다.
3. 지나치게 오래된 글은 밀린다.
4. 같은 주제만 반복되지 않게 분산한다.

### 8.4 Queue Rules

#### 오늘 읽을 글

- 안 읽은 글만 포함
- 관심사 점수 상위권
- 최신성도 강하게 반영
- 같은 카테고리/블로그 중복 제한

#### 관심 주제에서 고르기

- 관심 카테고리 상위 2~4개 산출
- 각 카테고리별 안 읽은 글 수 계산
- 카드보다 진입점 역할이 우선

#### 놓치기 쉬운 글

- 안 읽은 글만 포함
- 관심사 점수는 높지만 최신 상단에서는 밀린 글
- 최근성은 유지하되 `오늘 읽을 글`보다 느슨하게 적용

#### 최신 글

- 단순 최신순 미리보기
- 개인화보다는 보조 탐색 역할

### 8.5 Refresh Behavior

추천 새로고침은 랜덤 셔플이 아니다.

- `오늘 읽을 글`은 현재 노출된 카드 다음 우선순위 후보로 교체한다.
- `관심 주제에서 고르기`는 잦은 교체 대상이 아니다.
- `놓치기 쉬운 글`은 전체 새로고침 또는 데이터 갱신 시 재계산한다.

즉, 새로고침은 `다음으로 적절한 대안 제시`여야 하며, 우연한 랜덤 재추첨이 아니어야 한다.

## 9. Architecture Changes

### 9.1 Repository Layer

`ArticleRepository`는 계속 원본 기사 공급 역할을 맡는다.

- 기사 조회 책임은 유지
- 홈 추천용 후보 풀 조회 메서드 `getHomeFeedCandidates()`를 추가한다.
- 추천 계산 책임은 두지 않음

### 9.2 Interest Profile Layer

읽기이력과 즐겨찾기에서 관심사 프로필을 계산하는 레이어를 추가한다.

이 레이어의 기준 이름은 `HomeInterestProfileCalculator`로 한다.

이 레이어의 책임은 다음과 같다.

- 카테고리 선호도 계산
- 블로그 선호도 계산
- 최근 행동 가중치 반영

### 9.3 Home Feed Composer

홈 큐 조합 전담 레이어를 추가한다.

이 레이어의 기준 이름은 `HomeFeedComposer`로 한다.

입력:

- 후보 기사 목록
- 읽은 기사 ID
- 즐겨찾기 기사 목록
- 관심사 프로필

출력:

- `todayPicks`
- `interestTopics`
- `missedArticles`
- `latestPreview`

이 레이어는 홈용 큐를 조합하는 역할만 맡고, UI 상태나 저장소 접근을 직접 처리하지 않는다.

### 9.4 HomeViewModel

`HomeViewModel`의 역할은 다음 중심으로 축소한다.

- 홈 데이터 로딩 오케스트레이션
- 섹션 단위 로딩/갱신 상태 관리
- 사용자 액션 처리
- 스크롤 위치 복원

추천 세부 계산은 ViewModel 밖으로 이동한다.

### 9.5 HomeUiState

`HomeUiState`는 리스트 중심 구조에서 섹션 중심 구조로 바꾼다.

예시 필드:

- `todayPicks`
- `interestTopics`
- `missedArticles`
- `latestPreview`
- `isRefreshingTodayPicks`
- `isLoading`
- `error`
- `lastVisitTime`

기존 `recommendedArticles` 단일 필드는 제거 대상이다.

### 9.6 HomeScreen

홈 화면은 여전히 `LazyColumn`을 사용할 수 있지만, 단일 피드가 아니라 `섹션 렌더러`가 된다.

- 각 아이템은 기사 한 건보다 섹션 블록이 중심
- 홈 전용 섹션 컴포넌트를 추가한다.
- 기존 `ArticleCard`는 재사용하되, 큐 유형별 표현 차이를 허용

## 10. Existing Filter Treatment

현재 홈 상단 카테고리 필터는 `긴 리스트를 좁히는 도구` 성격이 강하다.

멀티 큐 홈에서는 `홈 상단 고정 카테고리 필터를 유지하지 않는다`를 기준으로 잡는다. 홈은 선택 허브이고, 필터형 리스트 화면은 별도 흐름으로 분리한다.

- 홈의 `관심 주제에서 고르기` 섹션이 빠른 관심 분야 진입점 역할을 맡는다.
- 카테고리 필터와 긴 최신 리스트는 `LatestFeedScreen`에 배치한다.

## 11. State Handling and Fallbacks

### 11.1 Cold Start

읽기이력과 즐겨찾기가 거의 없는 사용자에게는 개인화 실패를 감추려 하지 않는다. 대신 균형 잡힌 기본 큐를 제공한다.

기본 기준:

- 최근성 높은 기사
- 카테고리 다양성
- 블로그 분산

### 11.2 Empty States

큐 단위 빈 상태를 허용한다.

- 한 큐가 비어도 홈 전체는 정상 동작
- 상황에 맞는 대체 문구 또는 대체 섹션 사용
- 필요하면 특정 큐는 숨김

예:

- `오늘 읽을 글` 후보가 약하면 기본 추천 큐로 대체
- `놓치기 쉬운 글`이 없으면 섹션 숨김 또는 최신 미리보기 확장

### 11.3 Scroll Restore

상세 화면에서 돌아왔을 때 홈 섹션 상태와 스크롤 위치를 유지한다.

- 사용자가 어느 큐에서 들어갔는지 맥락이 깨지지 않아야 함
- 기존 스크롤 위치 저장 로직을 재사용하되, 섹션 중심 레이아웃에서 동일하게 동작하도록 검증한다.

### 11.4 Error Handling

오류는 전체 화면 실패보다 `부분 실패 허용`을 우선한다.

- 홈 전체 후보 기사 로드 실패 시만 전체 에러 상태
- 특정 큐 조합 실패는 다른 큐를 유지하고, 실패 큐만 숨기거나 기본 큐로 대체

## 12. Testing Strategy

가장 중요한 검증 대상은 UI보다 추천 조합 로직이다.

필수 테스트 범위:

- 읽기이력과 즐겨찾기 가중치가 의도대로 점수에 반영되는지
- 읽은 글이 `오늘 읽을 글`에 다시 들어오지 않는지
- 같은 카테고리/블로그로 추천이 과도하게 몰리지 않는지
- cold start에서 큐가 비정상적으로 비지 않는지
- 새로고침이 중복/반복 추천을 과하게 만들지 않는지
- `놓치기 쉬운 글`이 최신 최상단과 완전히 동일해지지 않는지

UI 검증 범위:

- 홈 섹션 순서와 노출 수
- 상세 복귀 시 스크롤/상태 유지
- 섹션별 빈 상태 문구

## 13. Success Criteria

이 개편의 성공 기준은 스크롤 길이가 아니다.

다음이 충족되어야 한다.

- 홈에서 빠르게 한 글을 고를 수 있다.
- 추천 섹션별 역할이 직관적으로 구분된다.
- 같은 주제만 반복되지 않는다.
- 추천이 랜덤보다 납득 가능하게 느껴진다.
- 홈 추천 실험이 `HomeViewModel` 비대화 없이 계속 가능하다.

## 14. Implementation Considerations

구현 단계에서 다음 사항을 우선 검토한다.

- 홈 후보 기사 풀 조회 범위와 비용
- 관심사 프로필 계산에 필요한 로컬 데이터 shape
